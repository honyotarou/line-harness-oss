## サブ 0 — 入口サマリ

**目的**: 誰向けのメニューか・成功指標・チャネル数を一文で揃える。

壁打ち質問（短く答えてよい）:

1. **業種・用途**（例: 美容外科、通知専用ボット、採用、店舗予約）
2. **いちばんの KPI**（例: 予約完了、問い合わせ、電話、資料請求）
3. **公式アカウントはいくつ**か（1 つ / 複数。複数なら **それぞれ同じメニューか**）
4. **避けたいこと**（例: 押し売り感、誤タップ、医療断定コピー）

**成果物（このステップ）**: 3〜5 行の「コンテキストメモ」。

---

## サブ 1 — レイアウト

**目的**: LINE が許す **フル（2500×1686）** か **ハーフ（2500×843）** と、**エリア数**を決める。

壁打ち:

1. **主 CTA は 1 つ**に絞れるか（例: 「予約」）。サブは最大いくつまでか。
2. ハーフで足りるか / フルが必要か（リンク数・視認性）。
3. チャットバー文言 `chatBarText`（下部に出る短いラベル案）。

**成果物**: レイアウト種別 + エリア数 + 各エリアの「ラベル（ユーザー向け）」リスト。

---

## サブ 2 — アクション割当

**目的**: 各エリアの `action.type` を決める。

| type | 使いどころ |
|------|------------|
| `uri` | LIFF、HTTPS、**tel:**（電話） |
| `postback` | Webhook で受けて Bot 返信（`data` 文字列を設計） |
| `message` | 定型文をユーザーが送信（キーワード自動返信と組み合わせ） |

壁打ち:

1. **予約**は `uri` で LIFF が正か（リッチメニュー主導線の場合）。
2. **電話**は `tel:` でよいか、営業時間外の案内は別導線か。
3. postback を使うなら **`data` の命名規則**（例: `menu=faq`）。

**成果物**: エリアごとに「type + 先（URL / data / text）」の表。

---

## サブ 3 — チャット連携

**目的**: **常時表示のリッチメニュー**と **会話（Flex・シナリオ）**の役割を被らせない。

確認項目:

1. 友だち追加直後は **Flex** で案内するか（例: `WELCOME_ANXIETY_FLOW`）。メニューと**同じ CTA をどう差別化**するか。
2. **リッチメニュー** = 「いつでも同じ場所」。**Flex** = 「文脈付きの一回限りに近い案内」。
3. タグ・シナリオで **メニュー切替**（`switch_rich_menu`）が要るか。

**成果物**: 役割分担を 5 行以内で記述。

---

## サブ 4 — JSON たたき台

**目的**: `RichMenuObject` に近い形の **ドラフト**（コピペのたたき台）。

- `size` / `selected` / `name` / `chatBarText` / `areas[]`
- 各 `area`: `bounds`（x, y, width, height）と `action`

**注意**:

- **合計が画像サイズ内**であること。座標は後でデザイナーが微調整してよいが、**重複なし・はみ出しなし**を確認。
- 正しい最終 JSON は **画像とピクセル合わせ**が必要。

**成果物**: コメント付き JSON ドラフト（`name` は管理用でユーザー非表示）。

---

## サブ 5 — 画像・ブランド

**目的**: デザイナー向けブリーフ。

1. **必須ピクセル**（フル or ハーフ）を明記。
2. **タップ領域**が指で押しやすいか（細すぎない）。
3. **色・フォント・トーン**は親 Step 1（ブランド）/ `design-tokens` と揃えるか。
4. ライセンス（写真・イラスト）の扱い。

**成果物**: 画像ブリーフ（箇条書き）+ ファイル名・納品形式（PNG 推奨）。

---

## サブ 6 — 運用・マルチアカウント

**目的**: 誰がいつ更新するか、**別ボット**への展開方法。

1. **デフォルトメニュー**（全員）で足りるか、**友だち別**リンクが要るか。
2. マルチアカウント時は **DB `line_accounts` + `lineAccountId` クエリ**で API を分ける（`docs/wiki/09-Rich-Menus.md`）。
3. **Staging / 本番**でチャネルが分かれている場合の手順メモ。

**成果物**: 運用ルール 3〜7 行。

---

## サブ 7 — 実装ハンドオフ

**目的**: 実装者が迷わない **API 順序**と **検証**。

Harness（認証: `Authorization: Bearer <API_KEY>`）の典型順:

1. `POST /api/rich-menus` — 本文に `lineAccountId` を含めてよいが **LINE には送られない**（Worker が除去）。
2. `POST /api/rich-menus/:id/image` — バイナリ or base64 JSON。
3. `POST /api/rich-menus/:id/default` — 全員に紐づくデフォルト。

検証:

- `GET /api/rich-menus` で一覧に出るか。
- **実機 LINE** の **1:1 トーク**で下部メニューが出るか（グループでは出ないことがある）。

**成果物**: チェックリスト（コピペ用）。
