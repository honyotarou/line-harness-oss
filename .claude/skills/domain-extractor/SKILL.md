# domain-extractor

業務知識を「ヒト/モノ/コト」フレームワークで対話的に整理し、ドメインオブジェクト候補を抽出するスキル。
増田亨著『現場で役立つシステム設計の原則』のDDD手法に基づく。

## トリガー

- `/domain-extractor` と入力されたとき

## 動作

### チーム構成（マルチエージェント）

スキル実行時、以下の3つの役割で分析を進める:

| 役割 | 担当 | 説明 |
|------|------|------|
| ファシリテーター | メインClaude | ユーザーとの対話を進行し、ヒアリングを主導する |
| 業務分析家 | サブエージェント | ヒト/モノ/コトの分類・分析を実行する |
| レビュアー | サブエージェント | 分析の矛盾・漏れをチェックし改善提案する |

### 5つの基本原則

分析全体を通じて以下の原則を守ること:

1. **「コト」起点で分析を開始する** — 業務イベント（注文、承認、発送など）を最初に特定し、そこから関連するヒト・モノを導出する
2. **小さく独立したドメインオブジェクトを作る** — 1つのオブジェクトが1つの責務を持つよう設計する
3. **業務用語をそのままコード化する** — ユビキタス言語を徹底し、技術用語への翻訳を避ける
4. **段階的に成長させる** — 最初から完璧を目指さず、対話を重ねて洗練する
5. **ビジネスルールを体系的に抽出する** — 暗黙のルールを明示的にし、ドメインオブジェクトに埋め込む

## ヒアリングワークフロー

### 開始時の案内

```
ドメイン抽出を開始します。

対象の業務・システムについて教えてください:
- どんな業務ですか？（例: ECサイトの受注管理、社内の経費精算）
- 主な登場人物は？（例: 顧客、管理者、配送担当）
- 解決したい課題は？

まずは業務の概要から聞かせてください。
```

### Phase 1: コト（業務イベント）の洗い出し

ユーザーから業務概要を聞いたら、「コト」を起点に分析を始める。

**ファシリテーターの質問例:**
- 「この業務で発生する出来事（イベント）を時系列で教えてください」
- 「その出来事が起きるきっかけ（トリガー）は何ですか？」
- 「その出来事の結果、何が変わりますか？」

**業務分析家（サブエージェント）の作業:**
Agent ツールを使い、以下の分析を実行する:
- ユーザーの回答から業務イベントを抽出
- 各イベントの前後関係・依存関係を整理
- イベントフロー図（テキストベース）を作成

**出力フォーマット:**
```
【コト一覧】
1. {イベント名} — {説明}
   トリガー: {何がきっかけで発生するか}
   結果: {何が変わるか}
   関連するヒト: {誰が関与するか}
   関連するモノ: {何が使われるか}
```

### Phase 2: ヒト（アクター/ロール）の特定

コトの分析結果から、関与する「ヒト」を深掘りする。

**質問例:**
- 「{イベント名}を実行するのは誰ですか？」
- 「承認や確認が必要な場合、誰が行いますか？」
- 「同じ人が複数の役割を兼ねることはありますか？」

**出力フォーマット:**
```
【ヒト一覧】
1. {ロール名} — {説明}
   責務: {何をする人か}
   関与するコト: {どのイベントに関わるか}
   権限: {何ができるか/できないか}
```

### Phase 3: モノ（業務で扱う対象物）の特定

コトとヒトの分析から、扱われる「モノ」を抽出する。

**質問例:**
- 「{イベント名}で作成・更新されるものは何ですか？」
- 「{モノ名}にはどんな属性（項目）がありますか？」
- 「{モノ名}の状態は変化しますか？どんな状態がありますか？」

**出力フォーマット:**
```
【モノ一覧】
1. {モノ名} — {説明}
   属性: {主要な項目}
   状態遷移: {状態の変化があれば}
   ビジネスルール: {制約・条件}
```

### Phase 4: レビューと矛盾チェック

**レビュアー（サブエージェント）の作業:**
Agent ツールを使い、以下を検証する:
- ヒト/モノ/コトの整合性チェック（コトに関与するヒトやモノが一覧に存在するか）
- 漏れの検出（言及されたがリストに未登録の概念はないか）
- 命名の一貫性チェック（同じ概念に異なる名前が使われていないか）
- ビジネスルールの矛盾チェック

**出力フォーマット:**
```
【レビュー結果】
- 整合性: {OK / 問題あり → 詳細}
- 漏れ候補: {追加で確認すべき概念}
- 命名の揺れ: {統一が必要な用語}
- ルール矛盾: {矛盾する制約}
```

レビュー結果をユーザーに提示し、修正・追加のヒアリングを行う。

### Phase 5: ドメインオブジェクト候補の導出

全フェーズの分析を統合し、ドメインオブジェクト候補を導出する。

## 成果物

最終的に以下の構成でMarkdownレポートを出力する。ファイル名は `domain-analysis-{対象業務名}.md` とする。

```markdown
# ドメイン分析レポート: {対象業務名}

## 1. 業務概要
{ヒアリングで得られた業務の全体像}

## 2. コト（業務イベント）一覧
{Phase 1 の結果}

## 3. ヒト（アクター/ロール）一覧
{Phase 2 の結果}

## 4. モノ（業務対象物）一覧
{Phase 3 の結果}

## 5. ドメインオブジェクト候補

### 値オブジェクト（Value Object）
| オブジェクト名 | 説明 | 属性 | ビジネスルール |
|---------------|------|------|--------------|

### エンティティ（Entity）
| オブジェクト名 | 説明 | 識別子 | 主要属性 | 状態遷移 |
|---------------|------|--------|---------|---------|

### ドメインイベント（Domain Event）
| イベント名 | トリガー | 発行元 | 影響範囲 |
|-----------|---------|--------|---------|

### 集約（Aggregate）
| 集約ルート | 含まれるエンティティ/VO | 不変条件 |
|-----------|----------------------|---------|

## 6. 用語集（ユビキタス言語）
| 業務用語 | 英語名（コード用） | 説明 |
|---------|-------------------|------|
```

## 注意事項

- 各フェーズで必ずユーザーに確認を取り、勝手に先に進まないこと
- 業務用語はユーザーが使った言葉をそのまま採用すること（技術用語に置き換えない）
- 不明点があれば必ず質問し、推測で埋めないこと
- レポート出力先はプロジェクトルート直下とする
